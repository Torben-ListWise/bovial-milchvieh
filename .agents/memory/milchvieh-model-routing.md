## Rule
`getModelForTask(taskType: ModelTaskType)` in `artifacts/api-server/src/lib/agent.ts` is the single source of truth for all model strings. Every other file must import and call it — never hardcode a model string.

## Model mapping (verified 2026-07-01)
- Haiku (`claude-haiku-4-5-20251001`): newsletter_generation, insights_summary, doc_categorization, follow_up_generation, benchmark_extraction
- Sonnet (`claude-sonnet-4-6`): chat_analysis_simple (max_tokens=2048, reduced toolset), chat_analysis
- Opus (`claude-opus-4-6`): chat_analysis_deep

## Escalation logic (per-turn, in runAgent loop)
- Turn 0: uses `initialTaskType` (from `classifyQuestion()` heuristic in analysisService.ts)
- Turn N+1: auto-escalates to Opus when: ask_farmer was used in any prior turn, OR >6 tool calls in last 2 turns, OR depthLevel="deep"
- depthLevel="quick" caps at Sonnet; depthLevel="deep" allows Opus from Turn 0
- `chat_analysis_simple` uses reduced toolset (no ask_farmer, calculate_investment, emit_chart) and max_tokens=2048

## DB migration caveat
drizzle-kit push fails with error code 42710 (role already exists) in this environment.
**Use raw SQL instead:** `psql "$DATABASE_URL" -c "ALTER TABLE x ADD COLUMN IF NOT EXISTS col type;"`

**Why:** The drizzle-kit push script tries to re-create the milchvieh_analyst role which already exists. The `--force` flag does not bypass this.

**How to apply:** Any future schema column additions must use the raw SQL approach above, not `pnpm --filter @workspace/db run push`.

## Generated type files that need manual updates
When adding fields to API responses, update ALL of:
1. `lib/api-zod/src/generated/api.ts` (Zod schemas)
2. `lib/api-client-react/src/generated/api.schemas.ts` (TypeScript interfaces)
3. `lib/api-client-react/dist/generated/api.schemas.d.ts` (compiled .d.ts)
4. `lib/api-zod/dist/generated/types/<typename>.d.ts` (compiled .d.ts)
These are generated by orval but not auto-regenerated — edit them by hand when the orval pipeline is not run.
